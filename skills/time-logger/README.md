# time-logger

Pulls from wherever you actually did work — Slack, Google Calendar, Claude Code
sessions, GitHub, and Granola meeting notes — into one combined daily file: raw context
per source, plus a draft "Potential Time Entries" section. It doesn't log anything for
you automatically; the output is meant to be handed to a downstream interactive agent
with access to the real time-logging system (e.g. Cronos-sync tooling), which reviews
it against the raw context and pushes the actual entries.

## How it works

Two stages, run by Claude Code inside this repo:

1. **Prefetch** — small agents (`.claude/agents/fetch-*-day.md`), one per integration,
   each pull one source for a target date and write a raw summary to
   `raw/{source}/YYYY-MM-DD.md`. They run in parallel and only for whichever
   integrations you've enabled. Adding a new integration later is just a new
   `fetch-*-day` agent writing to a new `raw/{source}/` directory — nothing else needs
   to change.
2. **Generate** — `generate-time-entry` discovers whatever `raw/*/YYYY-MM-DD.md` files
   exist for that date (no fixed list) and assembles them into one file: a `##` section
   per source with that source's raw content, followed by a `## Potential Time Entries`
   section with a synthesized draft. It writes its best guess with no interaction —
   nothing is final or uploaded at this stage.

Corrections you give it ("always skip standup", "that project is for Acme") get appended
to `user-preferences.md` so the same correction isn't needed twice.

## Install

```bash
npx skills add snowpackdata/snowpack-claude-skills --skill time-logger
```

This bundles six subagents (`.claude/agents/fetch-*-day.md`, `.claude/agents/generate-time-entry.md`)
that normally wouldn't be discoverable by Claude Code once nested inside an installed
skill folder — `SKILL.md`'s first step (run automatically, once) installs them to the
global `~/.claude/agents/` so they work regardless of which project you have open.
Output always goes to `~/repos/time_logs/`, not wherever you happen to be.

**Manual clone** also works, if you'd rather work inside the repo directly instead of
installing globally:

```bash
git clone https://github.com/snowpackdata/snowpack-claude-skills.git
cd snowpack-claude-skills/skills/time-logger
./setup.sh
```

Either way, say **"setup"** afterward (from any project, if installed via `npx`; from
`skills/time-logger` itself, if cloned manually) — it probes which integrations are
actually available on your machine (Claude sessions locally, `gh auth status` for
GitHub, and whether the Slack / Google Calendar / Granola MCP connectors are reachable),
asks which ones to enable, and writes the results to `~/repos/time_logs/capabilities.yml`.

## Usage

Say to Claude Code, from any project (or with `skills/time-logger` open directly, if you
cloned it manually):

- **"setup"** / **"configure"** — run or re-run the integration setup flow above.
- **"prefetch [date]"** — fetch raw data for a date from every enabled integration.
- **"log [date]"** / **"generate time entries for [date]"** — generate the entry from
  whatever raw data already exists (runs prefetch implicitly if needed).
- Just tell it corrections in plain language — "always skip X meeting", "this work is
  for Y client", "Z meeting is a standup" — and it writes the rule to
  `user-preferences.md` for next time.

## Integrations

| Integration | Requires |
|---|---|
| Claude Code sessions | Nothing — always on, reads `~/.claude/projects/` locally |
| GitHub | `gh` CLI installed and authenticated (`gh auth login`) |
| Slack | Slack MCP connector configured in Claude Code + your Slack member ID |
| Google Calendar | Google Calendar MCP connector configured in Claude Code |
| Granola | Granola MCP connector configured in Claude Code |

## Key paths

| Path | Purpose |
|---|---|
| `capabilities.yml` | Which integrations are enabled (gitignored, machine-specific — generated by "setup") |
| `user-preferences.md` | Standing corrections/context the generate agent reads every run (gitignored) |
| `raw/{source}/YYYY-MM-DD.md` | Prefetched source data per date, one directory per enabled integration (gitignored) |
| `time_logs/time_entries_YYYYMMDD.md` | Combined output per date — raw sections + Potential Time Entries draft (gitignored) |

Everything under `raw/` and `time_logs/`, plus your `capabilities.yml` and
`user-preferences.md`, is gitignored — this repo ships the agents and templates, never
your actual activity data.
